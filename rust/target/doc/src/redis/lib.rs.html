1//! redis-rs is a Rust implementation of a Redis client library.  It exposes
2//! a general purpose interface to Redis and also provides specific helpers for
3//! commonly used functionality.
4//!
5//! The crate is called `redis` and you can depend on it via cargo:
6//!
7//! ```ini
8//! [dependencies.redis]
9//! version = "*"
10//! ```
11//!
12//! If you want to use the git version:
13//!
14//! ```ini
15//! [dependencies.redis]
16//! git = "https://github.com/redis-rs/redis-rs.git"
17//! ```
18//!
19//! # Basic Operation
20//!
21//! redis-rs exposes two API levels: a low- and a high-level part.
22//! The high-level part does not expose all the functionality of redis and
23//! might take some liberties in how it speaks the protocol.  The low-level
24//! part of the API allows you to express any request on the redis level.
25//! You can fluently switch between both API levels at any point.
26//!
27//! ## Connection Handling
28//!
29//! For connecting to redis you can use a client object which then can produce
30//! actual connections.  Connections and clients as well as results of
31//! connections and clients are considered `ConnectionLike` objects and
32//! can be used anywhere a request is made.
33//!
34//! The full canonical way to get a connection is to create a client and
35//! to ask for a connection from it:
36//!
37//! ```rust,no_run
38//! extern crate redis;
39//!
40//! fn do_something() -> redis::RedisResult<()> {
41//!     let client = redis::Client::open("redis://127.0.0.1/")?;
42//!     let mut con = client.get_connection()?;
43//!
44//!     /* do something here */
45//!
46//!     Ok(())
47//! }
48//! ```
49//!
50//! ## Optional Features
51//!
52//! There are a few features defined that can enable additional functionality
53//! if so desired.  Some of them are turned on by default.
54//!
55//! * `acl`: enables acl support (enabled by default)
56//! * `aio`: enables async IO support (enabled by default)
57//! * `geospatial`: enables geospatial support (enabled by default)
58//! * `script`: enables script support (enabled by default)
59//! * `r2d2`: enables r2d2 connection pool support (optional)
60//! * `ahash`: enables ahash map/set support & uses ahash internally (+7-10% performance) (optional)
61//! * `cluster`: enables redis cluster support (optional)
62//! * `cluster-async`: enables async redis cluster support (optional)
63//! * `tokio-comp`: enables support for tokio (optional)
64//! * `connection-manager`: enables support for automatic reconnection (optional)
65//! * `keep-alive`: enables keep-alive option on socket by means of `socket2` crate (optional)
66//!
67//! ## Connection Parameters
68//!
69//! redis-rs knows different ways to define where a connection should
70//! go.  The parameter to `Client::open` needs to implement the
71//! `IntoConnectionInfo` trait of which there are three implementations:
72//!
73//! * string slices in `redis://` URL format.
74//! * URL objects from the redis-url crate.
75//! * `ConnectionInfo` objects.
76//!
77//! The URL format is `redis://[<username>][:<password>@]<hostname>[:port][/<db>]`
78//!
79//! If Unix socket support is available you can use a unix URL in this format:
80//!
81//! `redis+unix:///<path>[?db=<db>[&pass=<password>][&user=<username>]]`
82//!
83//! For compatibility with some other redis libraries, the "unix" scheme
84//! is also supported:
85//!
86//! `unix:///<path>[?db=<db>][&pass=<password>][&user=<username>]]`
87//!
88//! ## Executing Low-Level Commands
89//!
90//! To execute low-level commands you can use the `cmd` function which allows
91//! you to build redis requests.  Once you have configured a command object
92//! to your liking you can send a query into any `ConnectionLike` object:
93//!
94//! ```rust,no_run
95//! fn do_something(con: &mut redis::Connection) -> redis::RedisResult<()> {
96//!     let _ : () = redis::cmd("SET").arg("my_key").arg(42).query(con)?;
97//!     Ok(())
98//! }
99//! ```
100//!
101//! Upon querying the return value is a result object.  If you do not care
102//! about the actual return value (other than that it is not a failure)
103//! you can always type annotate it to the unit type `()`.
104//!
105//! Note that commands with a sub-command (like "MEMORY USAGE", "ACL WHOAMI",
106//! "LATENCY HISTORY", etc) must specify the sub-command as a separate `arg`:
107//!
108//! ```rust,no_run
109//! fn do_something(con: &mut redis::Connection) -> redis::RedisResult<usize> {
110//!     // This will result in a server error: "unknown command `MEMORY USAGE`"
111//!     // because "USAGE" is technically a sub-command of "MEMORY".
112//!     redis::cmd("MEMORY USAGE").arg("my_key").query(con)?;
113//!
114//!     // However, this will work as you'd expect
115//!     redis::cmd("MEMORY").arg("USAGE").arg("my_key").query(con)
116//! }
117//! ```
118//!
119//! ## Executing High-Level Commands
120//!
121//! The high-level interface is similar.  For it to become available you
122//! need to use the `Commands` trait in which case all `ConnectionLike`
123//! objects the library provides will also have high-level methods which
124//! make working with the protocol easier:
125//!
126//! ```rust,no_run
127//! extern crate redis;
128//! use redis::Commands;
129//!
130//! fn do_something(con: &mut redis::Connection) -> redis::RedisResult<()> {
131//!     let _ : () = con.set("my_key", 42)?;
132//!     Ok(())
133//! }
134//! ```
135//!
136//! Note that high-level commands are work in progress and many are still
137//! missing!
138//!
139//! ## Type Conversions
140//!
141//! Because redis inherently is mostly type-less and the protocol is not
142//! exactly friendly to developers, this library provides flexible support
143//! for casting values to the intended results.  This is driven through the `FromRedisValue` and `ToRedisArgs` traits.
144//!
145//! The `arg` method of the command will accept a wide range of types through
146//! the `ToRedisArgs` trait and the `query` method of a command can convert the
147//! value to what you expect the function to return through the `FromRedisValue`
148//! trait.  This is quite flexible and allows vectors, tuples, hashsets, hashmaps
149//! as well as optional values:
150//!
151//! ```rust,no_run
152//! # use redis::Commands;
153//! # use std::collections::{HashMap, HashSet};
154//! # fn do_something() -> redis::RedisResult<()> {
155//! # let client = redis::Client::open("redis://127.0.0.1/").unwrap();
156//! # let mut con = client.get_connection().unwrap();
157//! let count : i32 = con.get("my_counter")?;
158//! let count = con.get("my_counter").unwrap_or(0i32);
159//! let k : Option<String> = con.get("missing_key")?;
160//! let name : String = con.get("my_name")?;
161//! let bin : Vec<u8> = con.get("my_binary")?;
162//! let map : HashMap<String, i32> = con.hgetall("my_hash")?;
163//! let keys : Vec<String> = con.hkeys("my_hash")?;
164//! let mems : HashSet<i32> = con.smembers("my_set")?;
165//! let (k1, k2) : (String, String) = con.get(&["k1", "k2"])?;
166//! # Ok(())
167//! # }
168//! ```
169//!
170//! # Iteration Protocol
171//!
172//! In addition to sending a single query, iterators are also supported.  When
173//! used with regular bulk responses they don't give you much over querying and
174//! converting into a vector (both use a vector internally) but they can also
175//! be used with `SCAN` like commands in which case iteration will send more
176//! queries until the cursor is exhausted:
177//!
178//! ```rust,ignore
179//! # fn do_something() -> redis::RedisResult<()> {
180//! # let client = redis::Client::open("redis://127.0.0.1/").unwrap();
181//! # let mut con = client.get_connection().unwrap();
182//! let mut iter : redis::Iter<isize> = redis::cmd("SSCAN").arg("my_set")
183//!     .cursor_arg(0).clone().iter(&mut con)?;
184//! for x in iter {
185//!     // do something with the item
186//! }
187//! # Ok(()) }
188//! ```
189//!
190//! As you can see the cursor argument needs to be defined with `cursor_arg`
191//! instead of `arg` so that the library knows which argument needs updating
192//! as the query is run for more items.
193//!
194//! # Pipelining
195//!
196//! In addition to simple queries you can also send command pipelines.  This
197//! is provided through the `pipe` function.  It works very similar to sending
198//! individual commands but you can send more than one in one go.  This also
199//! allows you to ignore individual results so that matching on the end result
200//! is easier:
201//!
202//! ```rust,no_run
203//! # fn do_something() -> redis::RedisResult<()> {
204//! # let client = redis::Client::open("redis://127.0.0.1/").unwrap();
205//! # let mut con = client.get_connection().unwrap();
206//! let (k1, k2) : (i32, i32) = redis::pipe()
207//!     .cmd("SET").arg("key_1").arg(42).ignore()
208//!     .cmd("SET").arg("key_2").arg(43).ignore()
209//!     .cmd("GET").arg("key_1")
210//!     .cmd("GET").arg("key_2").query(&mut con)?;
211//! # Ok(()) }
212//! ```
213//!
214//! If you want the pipeline to be wrapped in a `MULTI`/`EXEC` block you can
215//! easily do that by switching the pipeline into `atomic` mode.  From the
216//! caller's point of view nothing changes, the pipeline itself will take
217//! care of the rest for you:
218//!
219//! ```rust,no_run
220//! # fn do_something() -> redis::RedisResult<()> {
221//! # let client = redis::Client::open("redis://127.0.0.1/").unwrap();
222//! # let mut con = client.get_connection().unwrap();
223//! let (k1, k2) : (i32, i32) = redis::pipe()
224//!     .atomic()
225//!     .cmd("SET").arg("key_1").arg(42).ignore()
226//!     .cmd("SET").arg("key_2").arg(43).ignore()
227//!     .cmd("GET").arg("key_1")
228//!     .cmd("GET").arg("key_2").query(&mut con)?;
229//! # Ok(()) }
230//! ```
231//!
232//! You can also use high-level commands on pipelines:
233//!
234//! ```rust,no_run
235//! # fn do_something() -> redis::RedisResult<()> {
236//! # let client = redis::Client::open("redis://127.0.0.1/").unwrap();
237//! # let mut con = client.get_connection().unwrap();
238//! let (k1, k2) : (i32, i32) = redis::pipe()
239//!     .atomic()
240//!     .set("key_1", 42).ignore()
241//!     .set("key_2", 43).ignore()
242//!     .get("key_1")
243//!     .get("key_2").query(&mut con)?;
244//! # Ok(()) }
245//! ```
246//!
247//! # Transactions
248//!
249//! Transactions are available through atomic pipelines.  In order to use
250//! them in a more simple way you can use the `transaction` function of a
251//! connection:
252//!
253//! ```rust,no_run
254//! # fn do_something() -> redis::RedisResult<()> {
255//! use redis::Commands;
256//! # let client = redis::Client::open("redis://127.0.0.1/").unwrap();
257//! # let mut con = client.get_connection().unwrap();
258//! let key = "the_key";
259//! let (new_val,) : (isize,) = redis::transaction(&mut con, &[key], |con, pipe| {
260//!     let old_val : isize = con.get(key)?;
261//!     pipe
262//!         .set(key, old_val + 1).ignore()
263//!         .get(key).query(con)
264//! })?;
265//! println!("The incremented number is: {}", new_val);
266//! # Ok(()) }
267//! ```
268//!
269//! For more information see the `transaction` function.
270//!
271//! # PubSub
272//!
273//! Pubsub is currently work in progress but provided through the `PubSub`
274//! connection object.  Due to the fact that Rust does not have support
275//! for async IO in libnative yet, the API does not provide a way to
276//! read messages with any form of timeout yet.
277//!
278//! Example usage:
279//!
280//! ```rust,no_run
281//! # fn do_something() -> redis::RedisResult<()> {
282//! let client = redis::Client::open("redis://127.0.0.1/")?;
283//! let mut con = client.get_connection()?;
284//! let mut pubsub = con.as_pubsub();
285//! pubsub.subscribe("channel_1")?;
286//! pubsub.subscribe("channel_2")?;
287//!
288//! loop {
289//!     let msg = pubsub.get_message()?;
290//!     let payload : String = msg.get_payload()?;
291//!     println!("channel '{}': {}", msg.get_channel_name(), payload);
292//! }
293//! # }
294//! ```
295//!

299# Scripts
300
301Lua scripts are supported through the `Script` type in a convenient
302way (it does not support pipelining currently).  It will automatically
303load the script if it does not exist and invoke it.
304
305Example:
306
307```rust,no_run
308# fn do_something() -> redis::RedisResult<()> {
309# let client = redis::Client::open("redis://127.0.0.1/").unwrap();
310# let mut con = client.get_connection().unwrap();
311let script = redis::Script::new(r"
312    return tonumber(ARGV[1]) + tonumber(ARGV[2]);
313");
314let result : isize = script.arg(1).arg(2).invoke(&mut con)?;
315assert_eq!(result, 3);
316# Ok(()) }
317```

324# Async
325
326In addition to the synchronous interface that's been explained above there also exists an
327asynchronous interface based on [`futures`][] and [`tokio`][].
328
329This interface exists under the `aio` (async io) module (which requires that the `aio` feature
330is enabled) and largely mirrors the synchronous with a few concessions to make it fit the
331constraints of `futures`.
332
333```rust,no_run
334use futures::prelude::*;
335use redis::AsyncCommands;
336
337# #[tokio::main]
338# async fn main() -> redis::RedisResult<()> {
339let client = redis::Client::open("redis://127.0.0.1/").unwrap();
340let mut con = client.get_async_connection().await?;
341
342con.set("key1", b"foo").await?;
343
344redis::cmd("SET").arg(&["key2", "bar"]).query_async(&mut con).await?;
345
346let result = redis::cmd("MGET")
347 .arg(&["key1", "key2"])
348 .query_async(&mut con)
349 .await;
350assert_eq!(result, Ok(("foo".to_string(), b"bar".to_vec())));
351# Ok(()) }
352```
